# PRD：从人工救火升级为自主质量运营

## 1. 真实基线

2026-07-12 的本地真实快照显示：

- 259 个来源最近检查中，127 个健康、59 个失败、28 个降级、45 个跳过；
- 261 个来源中只有 5 个 active，大量来源仍停留在 shadow；
- 来源雷达 70 条发现中有 50 条 pending；
- 星探只有 4 条已发布数据，生成器持续命中固定事件后被冷却期跳过；
- 评测为 57 分，价值和有效性维度没有真实用户反馈样本。

## 2. 产品目标

1. 每次来源审计后自动修复可恢复的抓取路径，并根据连续失败自动降级、隔离或停用不稳定来源。
2. 来源雷达自动完成去重、已有来源匹配和候选提案；许可或身份不清的来源只进入 draft，不自动激活。
3. 星探持续从新的高价值事件生成 inbox 建议，避免固定数据和重复主题。
4. 后台来源操作提供明确的可执行条件、运行中状态和失败原因，启用、隔离、单源拉取、观察模式都可验证。
5. 评测使用真实运行和反馈证据提升，Actions 持续记录分数和缺口；不得通过修改权重、伪造反馈或降低门槛制造 80 分。
6. 评测运行历史必须进入版本化仓库快照，最新评测必须形成可读报告；全新 clone 恢复数据后，不能只看到临时 Actions artifact 或空评测状态。

## 3. 自动化边界

自动执行可逆、可审计的动作：抓取、健康检查、观察模式、生命周期降级、候选归并、草稿提案、星探 inbox、质量评测和静态导出。

以下动作继续保留明确审批：

- 公开发布 Event；
- 将 Scout 建议发布为公开判断或触发外部动作；
- 许可、robots、身份或安全边界不明确的来源激活；
- 合并存在事实冲突的事件。

“全自动化”指系统把对象推进到最后一个必要决策点，并提供证据与建议，不是删除治理边界。

## 4. 验收目标

- 抓取器优先使用页面发现的 RSS/Atom，避免把无日期 HTML 卡片当成有效结果；
- 连续失败来源自动退出生产采集，恢复后经过 observation 再晋级；
- 来源雷达 pending 数持续下降，所有自动动作可追踪；
- 星探每轮从全部候选事件中寻找未处于冷却期的事件，数据可持续变化；
- 后台按钮不会重复提交，失败时展示具体阻塞条件；
- CI、来源审计和数据刷新形成独立且可恢复的 GitHub Actions 闭环；
- 评分报告明确区分当前分数、80 分目标、证据不足项和下一步，不把目标写成已达成。
- CI 从完整仓库快照重新评测，与上一提交的版本化报告比较；总分、证据覆盖或任一维度回退时失败，未达 80 分时持续输出按加权缺口排序的真实改进动作。
