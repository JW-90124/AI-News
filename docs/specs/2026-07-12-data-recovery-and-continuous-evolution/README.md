- 状态：实施中

- 目标版本：v0.5.x
- 前置规格：`2026-07-11-source-platform-and-scout`、`2026-07-11-two-year-intelligence-and-continuous-publish`

本规格处理 Agent Pulse 当前最关键的供给问题：来源目录较大，但有效观测、运行诊断、内容晋级和公开发布之间存在明显损耗。实施遵循“全部检查、分层修复、谨慎晋级、失败隔离、证据发布”，不把一次可访问或通用 HTML 抽取成功等同于生产可用。

## 五小时执行循环

```text
基线审计
  -> 来源检查与失败分类
  -> 修复最高价值故障簇
  -> 真实采集与质量门禁
  -> 事件/证据/发布损耗分析
  -> 管理台与公开页验收
  -> 记录指标和下一轮任务
  -> 重复
```

每轮必须留下可复现的输入、输出、指标、失败样本和下一轮选择理由。长时间运行不授权绕过登录、WAF、验证码、付费墙、robots 或来源许可。

## 2026-07-13 数据连续性增补

本轮把“能恢复”收紧为四条可验证的不变量：

1. 任何环境从仓库快照恢复时，不得用旧快照覆盖本地更新、更完整正文摘要、较新来源状态或既有运行历史；
2. 同一 canonical Signal 再次出现时按稳定身份合并，并把不同来源的发现保存为独立 observation，而不是静默丢弃；
3. 新来源继续经过统一 `SourceAdapter`、生命周期和合规门禁，只有可自动采集且未被限制的来源进入全量同步；
4. GitHub Actions 在生成快照前必须再次读取并合并 `origin/main` 的最新快照，随后才写出并提交，保证串行任务和人工更新都以集合并集为基础增长。

仓库快照只承担可公开、可审计的规范化数据与关系恢复，不承诺保存第三方完整正文或原始 payload。需要持久保存的私有原文仍应进入受控数据库或对象存储，永不提交到 GitHub Pages。
